# Amazon评论分析方法与质量标准

本文件是从 4 条校准和 40 条分层试编码中提炼出的全量分析规范，也是 559 条评论编码的唯一方法事实源；`05_calibration-coding.md` 和 `06_stratified-pilot-coding.jsonl` 只保留为校准证据与示例，不再承担规则定义。

当前状态：**559 条正式编码、机械全检、全量语义复核、固定主题重分配、全量业务建议和全量交互式报告均已完成。** `17–18` 直接消费 `08_full-review-coding.jsonl`、`12_full-theme-analysis.json` 和 `13_business-recommendations.md`，以 `14–16` 为报告结构模板。最近一次复核重新检查了 559 条 Review 的编码语义，并在保留既定 16 个产品主题、10 个用户主题及机会链定义的前提下刷新 `12` 的成员与计数；没有重新发现主题或重做业务建议框架。决策记录见 `../../../../docs/decisions/0001-full-review-coding-contract.md`，当前续工状态见 `../../../../handoff.md`。


## 目标与边界

评论分析的目标不是生成一个内容繁多的工作台，而是把评论证据转成可追溯、可比较、可行动的消费者洞察，最终服务三类决策：

- Listing 文案应准确表达、澄清或避免承诺什么。
- 主副图与 A+ 应展示哪些任务、场景、价值和能力边界。
- 产品应优先验证或迭代哪些体验、故障和未满足需求。

本阶段只分析评论能证明的内容。评论不能证明未购买者行为、沉默用户态度、订单占比、转化率或真实故障率；这些问题需要其他数据源。

输入唯一使用 `source-exports/2026-07-18/04_merged-reviews.jsonl`。英文标题和正文是证据事实，中文译文只帮助理解，不得替代英文证据或参与证据逐字校验。


## 分析方法

1. **先锁定 Review，再拆反馈点。** Review 层记录消费者、场景、任务、动机、预期、结果和价值链；反馈点层把一条评论拆成可以独立判断好坏、结果或需求的最小业务单元。
2. **先事实，后解释，再到动作。** `证据原文` 和 `事实判断`只写评论能直接支持的内容；`解释假设`允许提出可能机制，但不能进入事实频次；`候选业务动作`只是待全量证据复核的方向。
3. **先逐条编码，后全局归一。** 第一遍保留评论原本的具体含义；全部编码完成后，再统一同义维度和主题，避免过早套用固定主题而漏掉新信号。
4. **先全量证据，再形成建议。** 单条评论可以触发候选动作，但不能直接决定改 Listing 或改产品。正式建议必须注明评论分母、证据条数、星级与时间分布、代表性证据和适用边界。
5. **内容优化与产品问题分开。** 页面能澄清能力边界和使用方式，但不能用说明或图片包装替代可靠性、延迟、续航等产品修复。


## 核心编码规则


### `unknown`

- 评论没有表达，或证据不足以可靠判断时，写 `unknown`。
- 英文标题和正文都属于当前评论的直接证据；不得从星级、商品页、常识或其他评论补齐当前评论缺失的信息。
- `unknown` 不等于否定，也不等于没有需求；它只表示当前评论没有足够证据。
- 正向评论没有明确问题时，问题写 `unknown`；负向评论没有明确满意点时，满意点写 `unknown`。


### 场景状态

`使用场景`固定拆成四类，不得合并：

- `实际场景`：评论者或被描述使用者已经发生的行为。
- `计划场景`：未来准备使用，但尚未发生。
- `用户估计`：对人数、容量、效果或适用性的个人判断，尚未实测。
- `转述证据`：来自其他评论、他人或外部信息的说法，不是当前评论者亲历。

计划、估计和转述不得计入已验证功能或场景频次。


### 价值链

价值链固定按三层记录：

1. `产品特征或体验`
2. `用户任务结果`
3. `情绪关系社会价值`

每条链必须同时记录 `事实边界` 和 `证据强度`。评论只支持“双麦可同时使用”时，可以解释为“减少轮换、支持同时参与”，但不能无证据扩写成“祖辈与孙辈共同娱乐”。家庭角色、关系或社会价值只有在评论明确提供人物、行为或结果时，才能作为事实。


### 证据强度

- `高`：直接亲历、明确行为、具体故障、明确决策或可观察结果。
- `中`：表达明确，但缺少版本、条件、上下文或完整因果。
- `低`：用户估计、转述证据或分析者解释。
- `unknown`：没有足够证据判断。

证据强度衡量单条评论内的直接程度，不代表该现象在总体中的普遍性。


### 排除项

配送、Amazon 退款和非产品客服体验不进入 Review 编码、反馈点或主题统计。退货、换货、停止使用和转购属于产品决策结果，可以保留；渠道如何处理退货不保留。


## 全量编码契约

`06` 的语义结构保留，但全量文件不照搬试点元数据。建议正式产物命名为 `08_full-review-coding.jsonl`，每条源评论对应一行。


### 顶层结构

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `编码序号` | 整数 | 按 `04` 数据行从 1 到 559，机械生成 |
| `来源定位` | 对象 | 固定包含数据行、review_id、评论日期、评星、标题，全部从 `04` 注入 |
| `Review层编码` | 对象 | 固定使用下表字段和顺序 |
| `Review层推测` | 对象 | 固定包含 `购买动机` 与 `期望结果` 的独立谨慎推测，不得写回事实层 |
| `反馈点` | 对象数组 | 至少 1 个；按评论内部出现顺序编号 |

`试编码序号`、`样本角色`和`入样理由`只服务 40 条分层抽样，正式全量文件不保留。


### Review 层

字段顺序固定如下：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `消费者` | 字符串 | 已知角色与特征的简洁描述，缺失写 `unknown` |
| `使用场景` | 对象 | 固定四类场景，每类都是字符串数组 |
| `用户任务` | 字符串数组 | 没有证据写 `["unknown"]` |
| `购买动机` | 字符串数组 | 没有证据写 `["unknown"]` |
| `期望结果` | 字符串数组 | 没有证据写 `["unknown"]` |
| `实际结果` | 字符串数组 | 没有证据写 `["unknown"]` |
| `满意点` | 字符串数组 | 没有明确满意点写 `["unknown"]` |
| `价值链` | 对象数组 | 每个对象固定五个字段；未知层写 `unknown` |
| `未满足的需求` | 字符串数组 | 没有明确或可靠推导写 `["unknown"]` |
| `证据强度` | 对象 | 固定包含 `等级` 与 `说明` |
| `总体极性` | 枚举字符串 | 只允许五级受控词表 |
| `决策结果` | 字符串 | 退货、换货、转购、保留等；缺失写 `unknown` |

`总体极性`只允许：`正向`、`正向为主`、`正负混合`、`负向为主`、`负向`。情绪强度写进事实判断或结果，不再创造“强负向”“负向为主，含声音正向”等新标签。


### Review 层推测

`Review层推测` 只处理 `购买动机` 和 `期望结果`，每项固定包含 `处理结果`、`可能推测`、`依据` 和 `事实边界`。`处理结果` 只允许 `已有直接证据`、`可谨慎推测`、`证据不足`：已有直接证据时不得再生成推测；只有事实层为 `unknown` 且评论内存在足够依据时才可谨慎推测；证据不足时推测与依据均写 `["unknown"]`。推测不进入事实频次、主题覆盖或市场比例。


### 反馈点层

字段顺序固定如下：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `unit_id` | 字符串 | 由主流程按编码序号与单元序号机械生成 |
| `证据原文` | 字符串 | 必须是当前英文标题或正文的逐字连续子串 |
| `事实判断` | 字符串 | 忠实概括当前证据，不加入根因或外部事实 |
| `极性` | 枚举字符串 | 只允许 `正向`、`中性`、`负向`、`正负混合` |
| `维度` | 字符串数组 | 第一遍允许简洁开放标签，全量后再统一同义词 |
| `满意点` | 字符串数组 | 没有写 `["unknown"]` |
| `问题` | 字符串数组 | 没有写 `["unknown"]` |
| `未满足的需求` | 字符串数组 | 没有写 `["unknown"]` |
| `未知项` | 字符串数组 | 没有写 `["unknown"]` |
| `解释假设` | 字符串 | 没有可靠解释写 `unknown` |
| `候选业务动作` | 对象 | 固定包含 `Listing`、`图片与A+`、`产品`；不适用写 `unknown` |

反馈点的极性只描述该证据单元，不从 Review 星级继承。用户估计和转述状态记录在事实与场景边界中，不创造“正向估计”等极性标签。


## 全量提速方案


### 40 条为什么慢

40 条阶段同时发生了六件事：选择校准样本、摸索字段、逐条语义编码、人工抄录来源定位与证据、修复结构漂移、反复做来源校验。耗时主要来自**规则仍在变化和机械工作没有分离**，不能按这个速度线性推算 559 条。


### 推荐执行方式

1. **冻结契约。** 用户确认本文件后，559 条执行期间不再随意增加字段或改枚举；发现真正阻塞的问题时暂停并显式讨论。
2. **机械预填。** 主流程从 `04` 注入编码序号、来源定位和英文原文，语义 Agent 不手抄元数据、不修改源字段。
3. **固定小批次。** 559 条按每批 10 条拆成 56 批，最后一批 9 条。此前“59 个 10 条批次”的计算不准确。
4. **受控并行。** 使用 8 个固定 Agent 并行组成工作池，每个 Agent 连续处理多个批次，总共约 7 轮；不同时启动 56 个互不校准的 Agent。
5. **统一上下文。** 每个 Agent 使用同一份冻结契约和 4 条校准锚点，只返回 `Review层编码`、`Review层推测` 与 `反馈点` 的结构化结果。
6. **批后即验。** 每批返回后立即做 JSON、字段、类型、枚举和英文证据子串校验；只返工失败批次，不等 559 条结束后整体返工。
7. **主流程合并。** 主流程注入定位字段、生成 unit_id、按源行排序并合并成唯一正式文件。Agent 不直接改写最终文件。
8. **全局归一。** 559 条全部通过逐条门禁后，再统一维度同义词、形成主题和业务优先级；不在并行编码时让各 Agent自行发明全局分类体系。

批次结果只在执行期间暂存，不提交仓库；合并并校验后删除。正式逐条编码只交付 `08_full-review-coding.jsonl`，不同时生成 CSV、Excel 或永久批次文件。


### 为什么不使用 56 个 Agent 同时跑

- 启动、提示词和结果回收本身有固定开销。
- Agent 数越多，字段类型、标签措辞和价值链解释越容易漂移。
- 发现契约问题时，过多并发任务会同时产出同一种错误。
- 10 条小批次仍然保留，便于局部重试；并行度由固定的 8 个工作 Agent 控制。

因此，提速来自“批次并行 + 机械校验 + 局部返工”，不是单纯把 Agent 数推到最大。


### 实际执行记录

- 运行环境最多允许 6 个 Agent 同时工作，因此实际使用 6 个固定工作 Agent 完成 56 个批次；这是运行时容量偏差，不改变用户确认的 8-Agent 优选方案和数据契约。
- 一个停滞批次由替补 Agent 继续处理，其余批次按固定工作池回收；临时批次文件未进入仓库。
- 最终共 559 条 Review、2,546 个反馈点，全部通过 schema、类型、枚举、来源定位和英文证据逐字回查。
- 全量语义复核逐条检查 559 条 Review，并由独立复核流程裁决候选修改；最终修正 431 条 Review，再次通过全量机械校验。
- 2,546 个反馈点中，137 个证据只来自英文标题、2,396 个只来自英文正文、13 个同时出现在标题与正文；报告必须将标题与正文分栏完整展示，不能隐藏标题证据。


## 质量门禁


### 编码前

- `04` 必须恰好 559 行，每行 13 个字段。
- `04` SHA-256 必须保持 `6dd60229bdcfbb35165e521bfeceb3511488b99a908f209bb11ff55a120e820a`。
- 编码契约、枚举、字段类型和排除项必须已经由用户确认。


### 每批机械校验

- 每一行都是合法 JSON，字段名称、顺序和类型完全一致。
- 每个源数据行只出现一次，来源定位与 `04` 完全相同。
- 每条 Review 至少一个反馈点，unit_id 在最终文件内唯一。
- 每段 `证据原文`都是对应英文标题或正文的逐字连续子串，不允许省略号改写、空格归一或中文回译。
- 受控字段只能使用规定枚举，不增加额外字段。
- 渠道体验不进入编码。


### 最终机械校验

- 最终恰好 559 行，覆盖 `04` 第 1 至 559 行且没有重复。
- `04` 的内容与哈希没有变化。
- 所有批次通过相同 schema、类型、枚举和证据回查。
- 正式文件中不存在 `样本角色`、`入样理由`、批次号、Agent 名称或模型名称。


### 语义复核

机械校验只能证明结构和逐字可追溯，不能判断反馈点是否漏拆、极性是否判断错误、证据来自标题却在页面中被隐藏，或事实解释是否越界。正式语义门禁必须完整检查 559 条 Review，风险分层只用于安排复核顺序和第二轮重点：

- 第一轮逐条对照英文标题、英文正文、中文标题和中文正文，检查反馈点完整性、证据边界、事实判断、极性、维度及 Review 层编码。
- 第二轮优先复核 `低` 或 `unknown` 证据强度、长评论、星级与反馈点极性冲突、硬件改动、人数容量承诺和潜在误导声明。
- 候选修改由独立复核流程判断采纳、拒绝或改写，主流程只合并通过裁决的结构化结果。
- 合并后重新执行全量机械校验，并抽查标题独有证据、长评论、差评、正向评论中的负向反馈及设备故障描述。
- 复核重点是反馈点漏拆、标题证据遗漏、事实与解释越界、极性误判、场景状态混淆、价值链跳跃和业务动作夸大。


## 主题汇总与业务动作标准


### 双层主题归并

主题归并同时使用 `08_full-review-coding.jsonl` 的 Review 层与反馈点层，二者职责不同，不得只使用反馈点，也不得从反馈点重新推导消费者、场景或用户任务。

- **用户主题**：以 Review 层为事实源，归并消费者、四类使用场景、用户任务、购买动机、期望结果、实际结果、满意点、价值链、未满足需求、证据强度、总体极性和决策结果。
- **产品体验主题**：以反馈点层为事实源，归并证据原文、事实判断、极性、维度、满意点、问题和未满足需求。开放维度先统一到稳定的对象域与体验属性，每个反馈点只设置一个主产品主题，避免主题总量重复计算。
- **用户机会链**：先通过同一 `编码序号`连接 Review 层与其反馈点，再判断产品体验是否与该用户主题中的任务、期望、实际结果或未满足需求存在明确语义对应。仅在同一 Review 中共现、但与该机会链无关的反馈点不得纳入。结构固定为消费者与关系、场景、用户任务、期望结果、关键产品体验、实际结果与价值、主要阻碍、未满足需求和证据边界。

主题匹配只读取约定语义字段的值，不得序列化整条 JSON 后匹配字段名。产品主主题按“明确对象维度 → 其余维度 → 事实判断、满意点、问题和未满足需求 → 英文证据原文”逐级匹配；直接维度优先于泛化词，英文关键词使用词边界，避免 `system`、`light`、`box` 等子串误伤。

Review 层字段为 `unknown` 时保持 `unknown`，不得用其他评论、常识或分析者想象补齐。两层明显矛盾或疑似漏编时进入复核，不得静默覆盖任一层。计划场景、用户估计和转述证据分别统计，不得并入实际场景。

当 `08` 因全量语义复核发生变化时，`12` 不重新发现主题。固定沿用已经验收的 16 个产品体验主题、10 个用户主题和机会链定义，对全部 2,546 个反馈点与 559 条 Review 重新分配主题成员，再机械刷新计数、分布和证据引用。


### 40 条主题试点门禁

主题全量归并前，先复用 `06_stratified-pilot-coding.jsonl` 对应的 40 个数据行，但所有值从经过全量复核的 `08_full-review-coding.jsonl` 读取。该样本用于验证方法，不用于估算 559 条总体频率。

试点必须满足：

- 恰好覆盖既定 40 个数据行，且所有 Review 与反馈点都来自 `08`。
- 用户主题能够保留消费者、实际与计划场景、任务、动机、期望、结果和需求，而非退化为功能清单。
- 每个反馈点恰好归入一个主产品体验主题，主题内 Review 数与反馈点数分别统计。
- 用户机会链中的用户侧字段来自 Review 层，产品体验字段来自相同 Review 且与该任务或需求语义对应的反馈点；同 Review 共现不能单独作为关联依据。
- 每个主题保留可回查的 `编码序号`、数据行或 `unit_id`，代表性证据必须是 `08` 中的英文证据原文。
- `unknown`、计划场景、用户估计和转述证据不被升级为已知事实或实际发生。
- 试点经人工审核通过后，才生成 559 条全量主题产物；如结构或口径调整，先更新本方法事实源，再重跑试点。

逐条编码通过后，主题汇总至少同时报告：

- 涉及的唯一 Review 数和反馈点数。
- 在 559 条中的明确分母，不使用 40 条试点分布。
- 星级、时间和关键消费者或场景分布。
- 高、中、低证据强度构成。
- 正向满意点、负向问题和未满足需求的成对关系。
- 可回到英文原文的代表性证据。
- 已知事实、解释假设和仍需外部验证的问题。

业务动作按证据要求分三类：

- **Listing 文案**：先实时抓取当前页面，证明页面确有缺口或歧义后再改。
- **图片与 A+**：优先展示任务链、真实比例、操作步骤和能力边界，不把单条评论写成普遍承诺。
- **产品**：可靠性、延迟、续航、应用和容量问题需结合复现、规格、退货或售后数据，不因单条评论直接立项。


## 全量报告组装与验收

全量报告是对既有事实源的组装，不是第二轮语义分析。正式输入固定为 `08` 的 Review 与反馈点编码、`12` 的主题集合和 `13` 的业务建议；`14–16` 只提供已经确认的内容结构、数据契约和交互方式。

- **全量集合与展示样本分离。** 总览、发现和动作按钮必须显示完整关联 Review 数或精确反馈点数；每页 8 条只控制弹窗渲染，不得反写成证据总量。代表证据单独存放，且必须是完整证据集合的子集。
- **动作证据从主题单元重建。** 每项动作根据其映射的产品体验主题 `unit_id` 和机会链 `关联unit_id` 生成完整 `evidence_refs`；不得沿用 40 条试点中的静态引用。`source_review_rows` 表示关联主题覆盖范围，不能冒充精确动作证据。
- **校验集合相等，而非只查非空。** 对每项动作重新计算预期 `unit_id` 集合，并与实际 `evidence_refs` 做集合相等校验；同时检查 Review、反馈点和来源主题都可追溯，引用无重复，代表证据不超过 8 条。
- **检查分布异常。** 总览小节、发现链和动作的证据数量不得全部相同，也不得全部等于展示上限；动作精确证据必须覆盖 40 条试点以外的 Review。HTML 中不得残留把“8 条”写成总量的按钮文案。
- **真实浏览器验收。** 至少分别打开一个发现弹窗和一个动作弹窗，核对按钮总数、弹窗完整总数、当前页 8 条和翻页后的页码；同时检查页面非空、控制台无警告或错误。


## 已踩过的坑

1. **边编码边改 schema。** 结果是前后记录不一致并产生大面积返工。全量前必须冻结契约。
2. **人工复制来源信息。** 容易写错数据行、review_id 和标题。全量由脚本机械注入。
3. **证据原文做了润色或省略。** 省略号、空格变化和同义改写会导致无法逐字回查。证据必须从英文原文原样截取。
4. **渠道内容从旁路泄漏。** 即使反馈点已排除，渠道词仍可能留在入样理由或场景里。全量不再保留入样理由，并做全文件渠道扫描。
5. **极性使用自由文本。** 40 条出现了多种“正向为主，含……”标签，无法稳定聚合。全量改为受控五级 Review 极性和四级反馈点极性。
6. **同一字段有时是字符串、有时是数组。** 这会让 Agent、脚本和后续统计增加分支。全量固定字段类型，未知数组统一写 `["unknown"]`。
7. **价值链跨越证据。** 产品功能可以支持任务解释，但不能凭想象补人物关系和社会价值。每条价值链必须写事实边界与证据强度。
8. **把计划、估计或转述当成实测。** 这会制造虚假的人数容量、HDMI 或场景证据。四类场景状态必须分开。
9. **用星级替代反馈点极性。** 3 星可能有致命退货原因，5 星也可能包含明确限制。每个反馈点独立判断。
10. **把强烈情绪当外部事实。** `scam`、`ruined`等只能证明用户感受和信任受损，不能直接证明欺诈或技术根因。
11. **把候选动作当最终建议。** 单条证据只能触发待验证方向；最终优先级必须基于全量重复证据和外部验证。
12. **把分层试点当总体样本。** 40 条刻意平衡星级并加权负向评论，只能校准方法，不能估算频率。
13. **把并行度等同于 Agent 数。** 过多独立 Agent 会增加协调成本和口径漂移。固定工作池、小批次、批后校验更稳也更快。
14. **把展示上限当成证据总量。** 代表证据和弹窗每页可以限制为 8 条，但按钮与统计必须使用完整集合；否则全量报告会伪装成“每项都恰好 8 条”。
15. **刷新主题覆盖却保留试点引用。** 只更新 `source_review_rows`、不重建 `evidence_refs`，会让动作卡看似全量、实际仍只引用 40 条试点。动作证据必须从 `12` 的主题单元机械重建并校验集合相等。
16. **只检查存在和非空。** 这种校验能放过数量被截断、证据仍局限于试点等问题。必须增加数量分布、试点外覆盖、完整集合相等和浏览器交互四类门禁。
