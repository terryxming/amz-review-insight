## 目标与边界

评论分析的目标不是生成一个内容繁多的工作台，而是把评论证据转成可追溯、可比较、可行动的消费者洞察，最终服务三类决策：

- Listing 文案应准确表达、澄清或避免承诺什么。
- 主副图与 A+ 应展示哪些任务、场景、价值和能力边界。
- 产品应优先验证或迭代哪些体验、故障和未满足需求。

本阶段只分析评论能证明的内容。评论不能证明未购买者行为、沉默用户态度、订单占比、转化率或真实故障率；这些问题需要其他数据源。

输入唯一使用 `source-exports/2026-07-18/04_merged-reviews.jsonl`。英文标题和正文是证据事实，中文译文只帮助理解，不得替代英文证据或参与证据逐字校验。


## 分析方法

1. **先锁定 Review，再拆反馈点。** Review 层记录消费者、场景、任务、动机、预期、结果和价值链；反馈点层把一条评论拆成可以独立判断好坏、结果或需求的最小业务单元。
2. **先事实，后解释，再到动作。** `证据原文` 和 `事实判断`只写评论能直接支持的内容；`解释假设`允许提出可能机制，但不能进入事实频次；`候选业务动作`只是待全量证据复核的方向。
3. **先逐条编码，后全局归一。** 第一遍保留评论原本的具体含义；全部编码完成后，再统一同义维度和主题，避免过早套用固定主题而漏掉新信号。
4. **先全量证据，再形成建议。** 单条评论可以触发候选动作，但不能直接决定改 Listing 或改产品。正式建议必须注明评论分母、证据条数、星级与时间分布、代表性证据和适用边界。
5. **内容优化与产品问题分开。** 页面能澄清能力边界和使用方式，但不能用说明或图片包装替代可靠性、延迟、续航等产品修复。


## 核心编码规则


### `unknown`

- 评论没有表达，或证据不足以可靠判断时，写 `unknown`。
- 不从星级、标题、商品页、常识或其他评论补齐当前评论缺失的信息。
- `unknown` 不等于否定，也不等于没有需求；它只表示当前评论没有足够证据。
- 正向评论没有明确问题时，问题写 `unknown`；负向评论没有明确满意点时，满意点写 `unknown`。


### 场景状态

`使用场景`固定拆成四类，不得合并：

- `实际场景`：评论者或被描述使用者已经发生的行为。
- `计划场景`：未来准备使用，但尚未发生。
- `用户估计`：对人数、容量、效果或适用性的个人判断，尚未实测。
- `转述证据`：来自其他评论、他人或外部信息的说法，不是当前评论者亲历。

计划、估计和转述不得计入已验证功能或场景频次。


### 价值链

价值链固定按三层记录：

1. `产品特征或体验`
2. `用户任务结果`
3. `情绪关系社会价值`

每条链必须同时记录 `事实边界` 和 `证据强度`。评论只支持“双麦可同时使用”时，可以解释为“减少轮换、支持同时参与”，但不能无证据扩写成“祖辈与孙辈共同娱乐”。家庭角色、关系或社会价值只有在评论明确提供人物、行为或结果时，才能作为事实。


### 证据强度

- `高`：直接亲历、明确行为、具体故障、明确决策或可观察结果。
- `中`：表达明确，但缺少版本、条件、上下文或完整因果。
- `低`：用户估计、转述证据或分析者解释。
- `unknown`：没有足够证据判断。

证据强度衡量单条评论内的直接程度，不代表该现象在总体中的普遍性。


### 排除项

配送、Amazon 退款和非产品客服体验不进入 Review 编码、反馈点或主题统计。退货、换货、停止使用和转购属于产品决策结果，可以保留；渠道如何处理退货不保留。

`试编码序号`、`样本角色`和`入样理由`只服务 40 条分层抽样，正式全量文件不保留。

字段顺序固定如下：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `unit_id` | 字符串 | 由主流程按编码序号与单元序号机械生成 |
| `证据原文` | 字符串 | 必须是当前英文标题或正文的逐字连续子串 |
| `事实判断` | 字符串 | 忠实概括当前证据，不加入根因或外部事实 |
| `极性` | 枚举字符串 | 只允许 `正向`、`中性`、`负向`、`正负混合` |
| `维度` | 字符串数组 | 第一遍允许简洁开放标签，全量后再统一同义词 |
| `满意点` | 字符串数组 | 没有写 `["unknown"]` |
| `问题` | 字符串数组 | 没有写 `["unknown"]` |
| `未满足的需求` | 字符串数组 | 没有写 `["unknown"]` |
| `未知项` | 字符串数组 | 没有写 `["unknown"]` |
| `解释假设` | 字符串 | 没有可靠解释写 `unknown` |
| `候选业务动作` | 对象 | 固定包含 `Listing`、`图片与A+`、`产品`；不适用写 `unknown` |

反馈点的极性只描述该证据单元，不从 Review 星级继承。用户估计和转述状态记录在事实与场景边界中，不创造“正向估计”等极性标签。


## 全量提速方案


### 40 条为什么慢

40 条阶段同时发生了六件事：选择校准样本、摸索字段、逐条语义编码、人工抄录来源定位与证据、修复结构漂移、反复做来源校验。耗时主要来自**规则仍在变化和机械工作没有分离**，不能按这个速度线性推算 559 条。


### 推荐执行方式

1. **冻结契约。** 用户确认本文件后，559 条执行期间不再随意增加字段或改枚举；发现真正阻塞的问题时暂停并显式讨论。
2. **机械预填。** 主流程从 `04` 注入编码序号、来源定位和英文原文，语义 Agent 不手抄元数据、不修改源字段。
3. **固定小批次。** 559 条按每批 10 条拆成 56 批，最后一批 9 条。此前“59 个 10 条批次”的计算不准确。
4. **受控并行。** 使用 8 个固定 Agent 并行组成工作池，每个 Agent 连续处理多个批次，总共约 7 轮；不同时启动 56 个互不校准的 Agent。
5. **统一上下文。** 每个 Agent 使用同一份冻结契约和 4 条校准锚点，只返回 `Review层编码` 与 `反馈点` 的结构化结果。
6. **批后即验。** 每批返回后立即做 JSON、字段、类型、枚举和英文证据子串校验；只返工失败批次，不等 559 条结束后整体返工。
7. **主流程合并。** 主流程注入定位字段、生成 unit_id、按源行排序并合并成唯一正式文件。Agent 不直接改写最终文件。
8. **全局归一。** 559 条全部通过逐条门禁后，再统一维度同义词、形成主题和业务优先级；不在并行编码时让各 Agent自行发明全局分类体系。

批次结果只在执行期间暂存，不提交仓库；合并并校验后删除。正式逐条编码只交付 `08_full-review-coding.jsonl`，不同时生成 CSV、Excel 或永久批次文件。


### 为什么不使用 56 个 Agent 同时跑

- 启动、提示词和结果回收本身有固定开销。
- Agent 数越多，字段类型、标签措辞和价值链解释越容易漂移。
- 发现契约问题时，过多并发任务会同时产出同一种错误。
- 10 条小批次仍然保留，便于局部重试；并行度由固定的 8 个工作 Agent 控制。

因此，提速来自“批次并行 + 机械校验 + 局部返工”，不是单纯把 Agent 数推到最大。

### 最终机械校验

- 最终恰好 559 行，覆盖 `04` 第 1 至 559 行且没有重复。

### 语义复核

机械校验不能判断“有没有过度解释”。语义复核采用风险分层，而不是把 559 条再完整人工做一遍：

- 复核全部 `低` 或 `unknown` 证据强度记录。
- 复核正文长度前 10% 的长评论。
- 复核星级与编码极性明显冲突的评论。
- 复核所有涉及产品硬件改动、人数容量承诺和潜在误导声明的候选动作。
- 对其余评论按星级分层随机抽查 10%。
- 复核重点是反馈点漏拆、事实与解释越界、场景状态混淆、价值链跳跃和业务动作夸大。


## 主题汇总与业务动作标准

逐条编码通过后，主题汇总至少同时报告：

- 涉及的唯一 Review 数和反馈点数。
- 在 559 条中的明确分母，不使用 40 条试点分布。
- 星级、时间和关键消费者或场景分布。
- 高、中、低证据强度构成。
- 正向满意点、负向问题和未满足需求的成对关系。
- 可回到英文原文的代表性证据。
- 已知事实、解释假设和仍需外部验证的问题。

业务动作按证据要求分三类：

- **Listing 文案**：先实时抓取当前页面，证明页面确有缺口或歧义后再改。
- **图片与 A+**：优先展示任务链、真实比例、操作步骤和能力边界，不把单条评论写成普遍承诺。
- **产品**：可靠性、延迟、续航、应用和容量问题需结合复现、规格、退货或售后数据，不因单条评论直接立项。


## 已踩过的坑

1. **边编码边改 schema。** 结果是前后记录不一致并产生大面积返工。全量前必须冻结契约。
2. **人工复制来源信息。** 容易写错数据行、review_id 和标题。全量由脚本机械注入。
3. **证据原文做了润色或省略。** 省略号、空格变化和同义改写会导致无法逐字回查。证据必须从英文原文原样截取。
4. **渠道内容从旁路泄漏。** 即使反馈点已排除，渠道词仍可能留在入样理由或场景里。全量不再保留入样理由，并做全文件渠道扫描。
5. **极性使用自由文本。** 40 条出现了多种“正向为主，含……”标签，无法稳定聚合。全量改为受控五级 Review 极性和四级反馈点极性。
6. **同一字段有时是字符串、有时是数组。** 这会让 Agent、脚本和后续统计增加分支。全量固定字段类型，未知数组统一写 `["unknown"]`。
7. **价值链跨越证据。** 产品功能可以支持任务解释，但不能凭想象补人物关系和社会价值。每条价值链必须写事实边界与证据强度。
8. **把计划、估计或转述当成实测。** 这会制造虚假的人数容量、HDMI 或场景证据。四类场景状态必须分开。
9. **用星级替代反馈点极性。** 3 星可能有致命退货原因，5 星也可能包含明确限制。每个反馈点独立判断。
10. **把强烈情绪当外部事实。** `scam`、`ruined`等只能证明用户感受和信任受损，不能直接证明欺诈或技术根因。
11. **把候选动作当最终建议。** 单条证据只能触发待验证方向；最终优先级必须基于全量重复证据和外部验证。
12. **把分层试点当总体样本。** 40 条刻意平衡星级并加权负向评论，只能校准方法，不能估算频率。
13. **把并行度等同于 Agent 数。** 过多独立 Agent 会增加协调成本和口径漂移。固定工作池、小批次、批后校验更稳也更快。
