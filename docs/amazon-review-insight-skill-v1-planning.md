# Amazon 评论洞察 Skill v1.0.0 规划记录

状态：待固化为 Skill，当前只记录经过本轮试点验证的流程、盲点与认知边界。

本文件记录从 `B0CR1R7FKP` 试点反推 Amazon 评论洞察 SOP 的依据。它不承担评论、编码、主题或报告事实源角色；对应事实源仍分别是 `04_merged-reviews.jsonl`、`08_full-review-coding.jsonl`、`12_full-theme-analysis.json`、`13_business-recommendations.md`、`17_full-overview-report.json` 和 `18_full-overview-report.html`。

`v1.0.0` 当前能够宣称稳定的是流程契约，不是跨品类通用性。现有试点证明该流程能够把一个美国站卡拉 OK 机 ASIN 转成可信、可下钻、可行动的 VOC 报告，但尚未证明它能稳定处理不同品类、语言和数据条件下的商品。


## 目标

把 Amazon 评论证据转成可追溯、可比较、可行动的 VOC 洞察，服务以下决策：

- Listing 文案应准确表达、澄清或避免承诺什么。
- 主副图和 A+ 应展示哪些消费者、场景、任务、价值与能力边界。
- 产品应优先验证或迭代哪些体验、故障与未满足需求。

最终交付必须同时回答两个问题：

1. 从评论中看到了什么。
2. 从这些发现中可以推导出应该做什么。


## 已完成历程

1. 从模糊目标出发，把任务从“分析评论”收敛为服务 Listing、图片/A+ 和产品迭代的 VOC 洞察。
2. 从 Sorftime MCP 与领星 ERP MCP 获取评论，并由用户确认 Sorftime 返回的目标变体属性。
3. 盘点两侧字段，确定日期、星级、标题和正文为公共字段，只保留经确认的来源私有字段。
4. 先用英文日期、星级、标题和正文完成标准化、匹配及去重，再新增中文翻译，形成 559 条 `04` 评论事实源。
5. 用 4 条差异明显的评论校准字段、证据边界、反馈点拆分粒度、`unknown` 和价值链。
6. 用 40 条分层评论验证 schema、字段类型、受控极性、四类场景状态和候选业务动作。
7. 冻结正式编码契约，使用固定 Agent 工作池和 10 条小批次完成 559 条 Review 层与反馈点层编码。
8. 执行机械全检和全量语义复核，最终形成 559 条 Review 和 2,546 个反馈点；本轮语义复核修正了 431 条 Review。
9. 同时消费 Review 层和反馈点层，形成用户主题、产品体验主题和用户机会链；不得只用反馈点重新推导消费者、场景和任务。
10. 核对当时在线 Listing，以全量主题为依据形成文案、图片/A+ 和产品验证建议。
11. 先用 40 条校准报告内容结构、数据契约与交互，再把已完成的全量编码、主题和业务建议组装为 559 条报告。
12. 因标题证据隐藏、展示上限污染统计等问题执行全量审计，重新校验反馈点、刷新固定主题成员与计数，并重建报告证据引用。

经过本轮验证的主链路是：

> 原始证据 → 标准化评论 → Review 事实 → 最小反馈点 → 主题 → 洞察 → 业务动作 → 可追溯报告


## 已证明有效的原则

- 先冻结 Review，再拆反馈点。
- 先事实，后解释，再形成动作。
- 先逐条编码，后全局归一。
- 先用英文原文去重，再翻译。
- 英文标题和正文都是直接证据，中文只帮助理解。
- `unknown` 表示证据不足，不表示否定或没有需求。
- 实际场景、计划场景、用户估计和转述证据必须分开。
- Review 星级不能替代反馈点极性。
- Review 层与反馈点层承担不同职责，主题归并必须同时使用两层。
- 单条评论只能触发候选动作，不能直接决定改页面或改产品。
- Listing 优化、图片/A+ 优化和产品整改必须分开判断。
- 全量报告是对正式事实源的组装，不是第二轮语义分析。
- 主题覆盖、精确动作证据、代表证据和分页展示是不同口径。
- 机械 schema 校验不能替代全量语义复核和真实浏览器验收。


## 关键盲点

### 单 ASIN 过拟合

当前字段、主题、机会链和报告表达都只在卡拉 OK 机上经过完整验证。服装尺码、耗材复购、食品口味、工具耐久性等品类可能需要不同的编码重点。发布为通用 Skill 前，至少需要使用不同品类和不同数据来源组合做前向测试。

### 数据清洗尚不可重复执行

仓库没有保留从 `01_sorftime-reviews-live.json` 与 `02_lingxing-reviews-live.jsonl` 确定性重建 `04_merged-reviews.jsonl` 的完整标准化、匹配、去重和翻译脚本。当前 `04` 是可信事实源，但还不是可重复生成的事实源。

### 契约同步需要机械门禁

本轮仓库整理已把源字段数、`Review层推测`、2,546 个反馈点、刷新后的主题成员和报告路径同步到人类可读文档。此前这些信息曾分别停留在旧文档与当前代码中，说明仅靠人工维护仍会再次漂移。Skill 需要用机械检查保证字段契约、事实源计数、产物路径和交接摘要一致，代码也不能成为唯一的人类可读规则来源。

`14–16` 还暴露出另一类可复现性缺口：它们是当时已确认的 40 条校准快照，但试点生成器后来改为读取持续更新的 `08`，当前无法原样重建旧统计。历史快照不应被回写；正式 Skill 必须记录每份校准报告对应的输入版本或内容哈希。

### 报告生成器是产品专用实现

现有报告脚本硬编码了 ASIN、日期目录、卡拉 OK 主题、动作映射和结论文案，不能直接打包成通用 Skill。通用报告契约与产品专用结论必须分离。

### 全量语义正确的成本很高

559 条 Review 中有 431 条在全量复核中发生修正，证明 schema、受控词表和逐字证据校验不足以保障语义质量。Skill 必须显式安排语义复核预算，不能把一次 Agent 输出当成正式结果。

### 评论覆盖不能解释为市场比例

`154/559` 只能说明 154 条评论含相关证据，不能说明 28% 的购买者属于该人群。评论者偏差、Vine、VP、时间跨度、变体归属和未评论购买者都会影响解释。

### 业务成功标准尚未闭环

当前已经定义报告结构和证据质量的成功标准，但没有通过改版后的点击、转化、退货、售后或利润指标验证动作效果。VOC 能决定值得验证什么，不能单独证明动作有效。

### 首次主题发现与固定主题刷新尚未分流

本轮在 `08` 修正后沿用已验收主题定义刷新 `12` 是正确的；新 ASIN 第一次运行则必须经历主题发现、40 条校准和人工确认，不能复用当前产品的主题定义。

### 在线 Listing 缺少可复现快照

`13` 记录了页面核对日期，但没有保存机器可消费的页面事实快照。页面发生变化后，难以完整复现当时为什么提出某项动作。

### 空正文规则存在冲突

早期文档曾规定正文为空的记录不进入 VOC，但第 497 条正文为空、标题提供了合法证据并已完成编码。正式规则应是：标题和正文均为空时才不能做文本编码；标题可独立成为直接证据。


## 已知的已知

- `04` 是评论事实源，英文原文不得被翻译或分析流程改写。
- 去重必须发生在翻译之前，中文不参与匹配。
- Sorftime 无法确认子 ASIN 时，必须询问用户选择对应变体属性。
- Sorftime 数据量有限，不能默认代表完整历史评论。
- Review 层与反馈点层不能互相替代。
- `unknown` 表示当前评论证据不足，不表示否定。
- 计划场景、用户估计和转述证据不能计入真实发生。
- 星级不能替代反馈点极性。
- 主题覆盖、精确动作证据和代表证据是三种不同口径。
- 业务动作必须包含当前现象、VOC 依据、建议动作和验证门槛。
- 报告必须先完整回答消费者、场景、任务、动机、期待、结果、价值和需求，再进入行动建议。
- 证据下钻应使用弹窗，不能把用户强制带离当前阅读位置。
- 大证据集合必须分页渲染，展示上限不能进入统计口径。


## 已知的未知

- 只有 Sorftime 近一年 100 条数据时，哪些结论可以发布，哪些必须降级。
- 不同评论量对应的最低样本门槛、校准样本数和语义复核强度。
- 新品类是否需要新增、删除或重新定义 Review 层字段。
- 多语言评论应该先翻译，还是使用原语言直接编码后再翻译结论。
- 新 ASIN 的主题稳定标准和人工验收门槛。
- Skill 的时间、Token、Agent 并发和失败重试预算。
- Listing、退货、转化、客服和售后数据如何进入业务优先级。
- Skill 应安装在仓库还是用户 Skill 目录，以及采用自动触发还是用户手动调用。
- `v1.0.0` 的兼容范围、升级规则和旧产物迁移策略。


## 未知的已知

以下知识最初没有被明确写出，但在协作与返工中逐步显性化：

- 用户知道什么样的报告像真正的 VOC，但最初无法用 schema 表达；反复校准后表现为消费者、场景和任务必须先于功能主题。
- 用户知道运营人员何时需要原话、数量、证据和当前状况，但最初只表现为“工作台用起来别扭”。
- 用户能够识别正确子 ASIN 变体及业务动作是否符合真实运营语境，这是 Agent 无法只凭评论可靠获得的知识。
- Agent 知道事实源、数据契约、集合相等校验和浏览器验收的重要性，但早期没有把所有门禁一次性前置。
- 报告交互偏好已经成为隐性设计规范：弹窗下钻、少筛选、完整动作名称、数量带分母、精确反馈点筛选和分页展示，但尚未形成独立报告契约。
- “发现”和“行动”不是两个互不相关的章节，而是一条必须可回查的证据链；这个阅读原则最初存在于用户判断中，经过多轮样稿才被表达清楚。


## 未知的未知

未知的未知无法被完整列出；一旦能具体指出，就已经转化为已知的未知。当前只能记录需要防范的风险类别：

- MCP 字段、调用上限、权限和返回结构发生变化。
- Amazon 评论编辑、父子 ASIN 迁移和模板化评论产生新的错配方式。
- 新品类出现当前 schema 无法表达的任务、关系或价值类型。
- 模型升级导致相同契约下的反馈点拆分和主题粒度漂移。
- 评论量达到数千或数万后，单文件 HTML 的体积、浏览器内存和检索交互失效。
- 用户把评论证据覆盖率误读成购买者比例、故障率或市场份额。
- 业务团队只执行页面包装，忽略真正需要解决的产品问题。
- 报告影响产品和页面后，新评论的构成发生变化，形成反馈回路。
- 评论真实性、激励评论、刷评和幸存者偏差对主题产生未被察觉的影响。


## v1.0.0 前置门禁

在正式创建 Skill 之前，依次完成以下工作：

1. 统一源字段文档、分析方法、交接文档和 Python 校验器的数据契约。
2. 补齐可重复的数据标准化、英文去重、翻译合并和源数据校验脚本。
3. 将首次主题发现与已确认主题刷新拆成两个明确分支。
4. 将通用报告数据契约、HTML 模板和产品专用结论文案分离。
5. 为在线 Listing 保存带日期、来源和字段的机器可消费事实快照。
6. 定义只有 Sorftime、只有领星、两者同时可用三种输入分支。
7. 定义评论量、语言、变体不确定性和数据缺失时的降级策略。
8. 用至少一个不同品类 ASIN 和一个无领星数据的 ASIN 做前向测试。
9. 明确报告质量成功标准与业务实验成功标准的边界。
10. 通过 Skill 结构校验和无上下文前向测试后，再标记 `v1.0.0` 可用。


## 建议的 Skill 结构

Skill 应使用渐进披露，避免把全部历史和 schema 塞入 `SKILL.md`：

```text
amazon-review-insight/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── references/
│   ├── source-contract.md
│   ├── review-coding-contract.md
│   ├── theme-and-action-contract.md
│   ├── report-contract.md
│   └── quality-gates.md
├── scripts/
│   ├── normalize-merge-reviews.py
│   ├── validate-review-source.py
│   ├── review-coding-pipeline.py
│   ├── validate-theme-analysis.py
│   └── build-voc-report.py
└── assets/
    └── report-template.html
```

`SKILL.md` 只保留执行顺序、分支选择、停止条件和每一步完成标准。详细字段、枚举、示例和质量规则放入 `references/`；确定性、易错和重复执行的工作放入 `scripts/`；报告外观与交互放入 `assets/`。


## 当前结论

本轮成果足以作为 Skill `v1.0.0` 的主要案例和契约来源，但不能原样复制为通用 Skill。先整理仓库、修复契约漂移、补齐数据清洗可重复性，再开始 Skill 固化。这样得到的 `v1.0.0` 才代表一个可执行、可验证、可迁移的 SOP，而不只是对当前 ASIN 历程的复述。
