# 0001 全量评论编码契约

状态：已接受  
日期：2026-07-19


## 背景

4 条校准编码和 40 条分层试编码已经证明分析框架基本稳定，但试点文件仍存在极性自由文本和集合字段类型漂移。559 条全量并行编码需要先冻结不可逆的数据契约，否则会造成跨批次不一致和整体返工。

完整方法、字段定义和质量门禁的唯一事实源是 [`07_analysis-method-and-standards.md`](../../data-pilot/B0CR1R7FKP/analysis/2026-07-18/07_analysis-method-and-standards.md)。本记录只保存用户作出的选择，不复制完整规则。


## 决策

用户于 2026-07-19 明确确认：

1. 正式文件删除试编码专属的 `样本角色` 和 `入样理由`。
2. Review 层与反馈点层使用 `07` 定义的两套受控极性词表。
3. 集合字段统一为数组，无证据时使用 `["unknown"]`。
4. 使用 8 个固定 Agent、每批 10 条；临时分批文件不进入仓库历史。
5. 全量编码后先做机械全检和风险分层语义复核，再做主题归并和业务建议。


## 影响

- `08_full-review-coding.jsonl` 是逐条全量编码的唯一正式产物。
- 559 条执行期间不随意增加字段、修改类型或扩充极性枚举。
- 机械校验失败只返工对应批次；语义复核完成前不生成最终业务建议。
- `04_merged-reviews.jsonl` 继续作为评论证据唯一事实源，不受编码流程修改。


## 执行偏差记录

2026-07-19 实际运行环境最多允许 6 个 Agent 同时工作，因此本轮使用 6 个固定工作 Agent 完成 56 个批次。用户确认的 8-Agent 方案仍是有足够运行时容量时的优选执行方式；本次偏差只增加轮次数，不改变字段、枚举、批次大小、校验顺序或正式产物契约。
