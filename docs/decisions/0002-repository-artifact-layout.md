# 0002 仓库产物目录布局

状态：已接受
日期：2026-07-20


## 背景

试点过程中，源数据、分析产物、报告交付和辅助数据混放，导致产物编号、事实源角色与生成路径难以辨认。目录调整会影响脚本、文档链接和跨设备续工，因此需要固定布局。


## 决策

- `source-exports` 保存 `01–04` 的源导出、字段盘点与合并评论事实源。
- `analysis` 保存 `05–13` 的校准、逐条编码、主题分析与业务建议。
- `reports` 保存 `14–18` 的报告结构校准、结构化消费数据与 HTML 交付。
- `supporting-data` 保存不属于评论正式产物序列的辅助业务数据。
- `scripts` 保持扁平，不为当前五个脚本增加额外层级。
- 临时运行缓存与工具输出不进入仓库。

正式编号、文件路径和事实源角色继续由 [`artifact-index.md`](../../data-pilot/B0CR1R7FKP/artifact-index.md) 唯一维护。


## 影响

- 报告构建脚本必须把 `14–18` 写入 `reports/2026-07-18`。
- `09` 与 `10` 只保留已占用编号记录，不创建占位文件。
- `05`、`06`、`11` 与 `14–16` 作为方法和交互校准证据继续保留。
- `.playwright-cli`、`outputs` 与 Python 字节码缓存被清理并忽略。
