# Implementation Plan: 消费者洞察全量编码与工作台扩展

## Overview

基于 `B0CR1R7FKP` 的558条严格子ASIN历史评论，建立可复核的消费者洞察编码流程，并在标签体系稳定后复用于用户提供的竞品ASIN。分析按“关系角色／使用场景／用户任务／购买动机／期望结果／未满足需求”六个维度展开；关键词与规则只负责召回候选，最终洞察必须经过逐条复核并保留原文证据。


## Architecture Decisions

- 将评论主表作为原始事实源，候选集、人工复核集、洞察证据集依次派生，不回写原始评论。
- 将“显性证据／分析归纳／问题证据反推／未知”作为结论的证据性质，避免把推断伪装成用户原话。
- 将候选命中数与最终确认数分开；候选命中数不得进入工作台KPI。
- 将评论提及率与消费者占比区分；未明确提及的年龄、收入等人口属性保持未知。
- 先稳定自有ASIN标签体系，再对竞品应用相同口径，避免边分析边修改比较尺子。

## Task Details

### Task 1: 定义六维标签契约与证据等级

**Description:** 固定六个分析维度的含义、允许值结构、证据性质和未知处理方式，为自有与竞品建立同一分析尺子。

**Acceptance criteria:**

- [x] 六个维度互不替代，同一评论允许多标签
- [x] 每个标签包含证据性质、原文片段和复核状态
- [x] 人口属性未显性出现时保持未知

**Verification:** 人工审阅家庭共同娱乐试点，确认现有11条主题证据能映射到新契约且不改变原口径。

**Dependencies:** None

**Estimated scope:** Small


### Task 2: 建立消费者主题候选发现脚本

**Description:** 扫描558条评论，根据可审计规则生成六维候选记录，供逐条复核，不直接判断主题成立。

**Acceptance criteria:**

- [x] 每条候选包含 `review_id`、维度、候选标签、规则标识和命中文本
- [x] 同一评论可命中多个维度，重复规则命中不重复生成相同候选
- [x] 输出记录统一标记 `review_status: pending`

**Verification:** 先写失败测试，再验证固定样本能召回家庭、聚会、双麦选择理由和麦克风问题等候选。

**Dependencies:** Task 1

**Estimated scope:** Medium


### Task 3: 验证候选脚本只做召回、不生成最终KPI

**Description:** 对宽关键词误报建立测试与文档边界，禁止候选数量进入洞察页面。

**Acceptance criteria:**

- [x] 候选输出不包含 `validated_count`、发生率或消费者占比字段
- [x] 未人工复核主题不能导出为真实洞察
- [x] 宽规则误报样本被保留为测试用例

**Verification:** 自动测试检查输出字段，并以“app”“battery”“bought”等宽词样本证明候选不等于结论。

**Dependencies:** Task 2

**Estimated scope:** Small


## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| 用户通常不会完整说明购买前动机 | 高 | 只将明确的购买、选择、替换或送礼表达编码为显性动机，其余标记为分析推断 |
| 关键词同时命中正向、负向和功能描述 | 高 | 候选逐条复核，不把命中数当KPI |
| 同一评论包含多位使用者和多个场景 | 中 | 使用多标签，不强制归入单一画像 |
| A版与D版评论无法可靠区分 | 高 | 保持版本未知，不将时间趋势归因于版本变化 |
| 标签体系在竞品阶段发生漂移 | 中 | 先冻结自有ASIN第一版定义，再以变更记录管理新增标签 |


## Open Questions

- 完整复核后，任务链是按“场景”还是按“核心任务”作为一级聚合对象，由真实主题结构决定，不预先锁死。
- 竞品缺少相同价格带时，比较重点可能需要从直接替代转为“相同使用场景下的低价方案”，待竞品评论数据验证。
